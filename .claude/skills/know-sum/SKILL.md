# know-sum — knowledge.md → answer.md（三段式认知生成）

从 `knowledge.md` 中提取核心知识，生成三段式认知文档 `answer.md`。

核心任务：把 knowledge.md 的结构化笔记转化为"它像什么？→ 但它不完全是…… → 它到底是什么？"的认知递进结构。

## 工作台结构

```
workbench/
├── answer.md              # 三段式回答文档（输出）
└── documents/
    └── knowledge.md       # 知识库（只读输入）
```

- 只能读取 `documents/knowledge.md`
- 只能修改 `answer.md`
- 如果 `knowledge.md` 不存在或为空，提示用户知识库尚未初始化，需要先提供文档进行学习

## 工作流程

### 1. 读取 knowledge.md

确认 `documents/knowledge.md` 存在且不为空，然后完整加载。

- 从元信息中了解已学习文档列表和**全局关键词索引**
- 浏览所有文档章节的标题和速读总结，了解知识覆盖范围

### 2. 确定主题

根据用户是否提供了问题/关键词，采用不同的策略：

**用户提供了问题/关键词**（如 "总结一下 TCP 流量控制"）：
- 在 knowledge.md 中搜索与用户问题**最相关的章节**
- 关键词匹配优先，同时考虑语义相关性
- 如果 knowledge.md 中有多个相关章节，以描述最详细的章节为主，其他章节作为补充

**用户没有提供问题**（如 "帮我总结一下"）：
- 从 knowledge.md 元信息的**"全局关键词索引"**中选取候选
- 选取标准（按优先级）：
  1. 在多个文档章节中**重复出现**的关键词
  2. 在 knowledge.md 中的**描述最详细**（对应章节最长、精读笔记最丰富）
  3. 在全局关键词索引中排位靠前
- 选定关键词后，定位到对应章节

### 3. 生成三段式内容

针对确定的主题概念，逐段生成：

#### 第一步："它像什么？"

从 knowledge.md 中找到概念的本质特征，设计一个日常生活中的类比：

- **类比场景**：用故事或场景描述，让完全不懂的人也能建立直觉
- **对应关系**：逐个映射场景元素到技术概念。格式：`**[场景元素 A]** ⇔ [概念中的 X]`

设计类比的要点：
- 类比必须基于 knowledge.md 中的核心特征，不能凭空编造
- 覆盖概念的**核心特征**而非边缘细节
- 类比天然不精确——它的作用是建立一个可修正的直觉框架

#### 第二步："但它不完全是……"

参考第一步的类比场景，用 knowledge.md 的精确知识修正类比中的认知偏差。

**修正条目的生成规则**（每条必须严格遵守）：
- **左列（"你可能会觉得…"）**：必须是类比中的**具体场景元素**引发的误解。从类比场景中提炼出一个可能让读者"想当然"的结论
- **右列（"实际上的精确真相是…"）**：必须来自 knowledge.md 中的**精确描述**（精读笔记或速读总结），用严谨但可读的语言纠正左列的偏差
- 覆盖四类典型偏差：**时序**（顺序与并发）、**角色**（谁做了什么）、**范围**（边界与适用条件）、**机制**（内部如何实现）
- 建议 3-5 行

格式示例：
```
| 三次握手是为了"确认双方都在线" | 它的首要目的是同步初始序列号（ISN），顺便验证对方在线。`tcp.txt §3.4` |
```

最后用**一句话**总结核心逻辑。

#### 第三步："它到底是什么？"

直接从 knowledge.md 提取精确的专业定义和解释：

- **专业定义**：引用 knowledge.md 中的标准定义或精读笔记中的核心描述（标注来源章节）
- **人话拆解**：将专业术语逐一用大白话解释。格式：`**"[术语]"** — 就是[通俗解释]`
- **关系**：如果拆解了多个术语，追加一个小节说明它们之间的关系。例如："A 是 B 的前提条件，C 在 A 和 B 的基础上提供了 X 能力"
- **为什么要有这个`概念`？**：从 knowledge.md 中提取文档的设计动机、存在意义

### 4. 写入 answer.md

按模板（见下方）组装并写入 `answer.md`。

如果 `answer.md` 已存在且包含其他概念的三段式，则覆盖（answer.md 一次只呈现一组核心概念）。如果 `answer.md` 是之前反复修正过的版本，保留其中的"但它不完全是……"修正条目（这些是用户积累的认知轨迹），只更新第一步和第三步的内容。

### 5. 输出与总结

生成 `answer.md` 后，向用户展示：

1. **聚焦的概念** — 明确指出本次三段式围绕哪个概念展开
2. **一句话本质** — 用一句话讲清楚这个概念到底解决了什么问题
3. **knowledge.md 中未纳入 answer.md 的内容** — 浏览 knowledge.md 所有章节，告知用户还有哪些文档/章节/知识点在 knowledge.md 中但未出现在 answer.md 中。让用户知道"学了哪些"和"还有哪些可以深挖"

## answer.md 模板

```markdown
# 理解 `[关键词]`
> [其他相关词汇，逗号分隔]

## 一句话抓住本质
[用一句话讲清它到底解决什么问题，或它的独特价值]

## 第一步："它像什么？"

### 类比场景
[用故事或场景描述]

### 对应关系
- **[场景元素 A]** ⇔ [概念中的 X]
- **[场景元素 B]** ⇔ [概念中的 Y]
- ...

---

## 第二步："但它不完全是……"

| 你可能会觉得… | 实际上的精确真相是… |
| :--- | :--- |
| [类比场景元素导致的误解] | [knowledge.md 中的精确解释] |
| [另一个类比偏差] | [knowledge.md 中的精确解释] |

> **一句话**：让你看到 [最核心的逻辑]，本质上是 [一句话说清它是什么运算/机制/规则]。

---

## 第三步："它到底是什么？"

### 专业定义
> [标准定义，引用自 knowledge.md `文档路径 §章节`]

### 人话拆解
- **"[术语1]"** — 就是 [通俗解释]
- **"[术语2]"** — 可以理解为 [通俗解释]

### 关系
[术语之间的关联——哪个是前提、哪个是结果、哪个增强哪个]

### 为什么要有`这个概念`？
[用一两句话说明它的设计动机、存在意义、解决了什么问题]

## 常见误解快查（可选）
- ❌ "……" → ✅ 实际是 "……"
```

## 约束

| 项目 | 规则 | 原因 |
| :--- | :--- | :--- |
| 前提条件 | `knowledge.md` 必须存在且不为空，否则提示知识库尚未初始化 | 没有知识库无法生成认知文档 |
| 知识来源 | 只从 `knowledge.md` 读取，不重新扫描原始文档 | 避免重复工作，保证知识一致性 |
| 主题选择 | 用户有提问→搜索最相关章节；无提问→从全局关键词索引选最高频 | 有针对性回答优先，否则自动找最有价值的概念 |
| 修正条目 | 左列必须是类比场景具体元素引发的误解，右列必须是 knowledge.md 中的精确描述 | 类比→偏差→修正的链条必须可追溯 |
| 修正覆盖 | 至少覆盖时序、角色、范围、机制四类偏差中的 3 类 | 类比最容易在这四个维度产生误导 |
| 权限范围 | 只写 `answer.md`，不修改 `knowledge.md` | 知识库由文档阅读流程维护 |
| 生成后输出 | 必须告知用户：聚焦概念 + 一句话本质 + knowledge.md 中未纳入的内容 | 用户需要知道学了什么、还有什么可学 |
