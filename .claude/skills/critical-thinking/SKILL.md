# 批判性思维分析框架

用三步认知流程帮用户系统性地思考：先追溯"它为什么是这样"（溯因），再思考"如果重来会怎样"（重构），最后验证"新方案真的更好吗"（对比）。

## 核心原则

- **追溯原因，不论合理或不合理**：每个设计背后都有原因——可能是技术限制、成本约束、深思熟虑后的最优解、也可能仅仅是惯性。我们的任务是找出真正的原因，不预设结论。
- **从第一性原理出发**：先拆解本质需求，再构建解决方案。不依赖类比和经验。
- **批判对象是设计，不是设计者**：目标是找到更好的方案，不是证明别人错了。如果现状确实是当时条件下的最优解，就说它合理。
- **默认轻量，按需深入**：大多数时候用户只是需要一个清晰的判断，不是长篇报告。先给出结论和核心理由，用户追问后才展开。

## 深度控制

**默认为 L1 轻量分析**：给出判断 + 核心理由（3-5 句），不生成正式报告。以对话的方式自然回应。

只有在以下情况才升级到 L2/L3：
- 用户明确要求深入分析
- 问题本身是架构决策/技术战略级别，表面回答会产生误导
- 用户在你的 L1 回复后继续追问

| 量级 | 触发条件 | 输出形式 |
|:---|:---|:---|
| **L1（默认）** | 所有分析类问题 | 对话式：一句话判断 + 核心理由 |
| **L2 标准** | 用户要求深入 / 问题足够复杂 | 三步精简报告 |
| **L3 深度** | 用户要求最详细 / 系统级决策 | 完整报告 + 多方案对比 |

**原则**：用户问一句就回一句。用户展开讨论才深入。不要对"为什么 TCP 要三次握手"输出 5000 字分析报告。

## 信息策略：什么时候问，什么时候答

这是整个框架最容易出错的地方。AI 无法感知自己"不知道什么"，所以需要明确的锚点来区分两种场景。

### L1：基于通用知识回答，容忍简化假设

L1 的目标是**快速给出一个有用的判断**，不是给出一个在任意前提下都成立的完备分析。

- 基于通用知识回答（设计原理、技术权衡、历史背景），不需要特定上下文。
- 允许做合理的简化假设。例如用户问"为什么 Redis 用单线程？"，可以假设 ta 在问经典版本的设计动机，直接回答"避免锁竞争 + 内存带宽不是瓶颈 + 简化实现"。
- **不要**在 L1 中追问用户"你是否了解 X？你的场景是 Y 还是 Z？"
- 回答末尾加一句免责："如果你有特定的上下文（比如不同的版本/场景/约束），结论可能不同，可以告诉我更多细节。"

### L2/L3：信息不足时停止，不要勉强

L2/L3 的目标是给出一个**经得起推敲的完整分析**。如果缺少关键信息，这个目标无法达成。

判断"信息是否足够"的锚点：**你能否回答 Step 1 的"追问起源"？** 如果不知道这个设计产生的上下文（技术背景、约束条件、目标用户），就不知道它的"合理性"建立在什么前提上。此时：

- 不要强行输出三步报告。
- 直接告诉用户："我需要知道 [X] 才能做有意义的分析。你能提供吗？"

**需要问用户的关键信息**（这个列表有限且明确）：
- 约束条件：时间/预算/团队规模/技术栈限制
- 目标用户：谁在用？什么场景？
- 设计动机：有没有原始的设计文档或决策记录？
- 规模：多大流量/多少用户/什么量级？

如果用户也答不上来，就基于最可能的假设做分析，并显式声明假设——这比沉默地猜要好。

## 三步框架

三步是**思考过程**，不是输出模板。在 L1 中，三步发生在你的思考里，只输出结论。在 L2/L3 中，三步映射到报告结构。

### Step 1：溯因分析 — 它为什么是这样？

从三个角度追问，目标是理解，不是挑刺：

1. **追问起源**：它最初被设计成这样的原因是什么？技术限制？成本约束？历史惯性？还是当时条件下的最优解？哪些当时的前提条件现在已经变了？

2. **剥离表象**：抛开当前实现的外壳，它本质上在解决什么核心问题？用一句话表达。

3. **寻找矛盾**：看三个地方——**冗余**（是否为了解决简单问题引入了不必要的复杂性？）、**摩擦**（使用者是否感到别扭？）、**副作用**（解决了 A 但引发了 B？）。

这一步结束时给出明确判断：
- **合理**：当时条件下的最优解，前提条件未变 → 不需要改
- **有历史原因但前提已变**：当年合理，现在条件变了 → 值得重新审视
- **不合理**：设计本身就有问题，即使在当时也不是最优 → 需要改

### Step 2：归零重构 — 如果重来会怎样？

只有当 Step 1 判断为"前提已变"或"不合理"时才展开。如果 Step 1 判为"合理"，跳过此步，只给 1-2 句优化方向（如果有的话）。

1. **拆解本质需求**：把核心诉求拆到不可再分的原子级。
2. **脑暴替代路径**：思考 2-3 种根本不同的实现路径。关键是**不一样**，不是微调。
3. **引入外部变量**：现在的技术/认知相比这个设计被创建时有什么飞跃？

### Step 3：对比验证 — 新方案真的更好吗？

1. **效率与成本**：更快/更便宜/更容易维护吗？收益是否大于替换成本（迁移、学习、生态重建）？
2. **体验的质变**：是否消除了用户真正的痛点？区分"设计者的审美"和"使用者的实际痛苦"。
3. **系统自洽性**：会不会引发连锁反应？解决了 A 但引发了更严重的 B？

最终判断：**值得替换 / 特定条件下值得替换 / 不值得替换但可局部优化 / 现状即最佳**。

**重要**：Step 3 的结论必须回扣 Step 1 的定性。如果 Step 1 判了"合理"，Step 3 就不能判"值得替换"。

## L2 报告模板

```
# [主题]

> **判断**：[一句话结论]

## 溯因 — 它为什么是这样？
[起源 + 本质问题 + 矛盾]

## 重构 — 如果重来
[原子级需求 + 2-3 个替代路径]

## 对比 — 新方案更好吗
[效率/体验/自洽性 → 最终推荐]
```

模板是指导，不强制填满每个子维度。

## 示例

**问题**：Dockerfile 的 `RUN` 指令默认用 `/bin/sh -c` 而不是直接执行，这个设计合理吗？

```
# Dockerfile RUN 指令为什么用 shell 包装？

> **判断**：有历史原因但前提已变——2013 年的合理选择，今天应该引导用户选择更安全的 exec form。

## 溯因 — 它为什么是这样？

Docker 推出时（2013），目标用户的心智模型是"在容器里跑 shell 命令"。用 `/bin/sh -c`
包装 RUN 降低了上手门槛——用户直接写熟悉的 shell 命令，不需要理解 exec 语义。

但前提条件已经变了：
1. 容器从开发工具变成了生产基础设施，信号处理的正确性变得至关重要。
   shell form 启动 sh 作为 PID1，不转发信号——"容器无法优雅停止"的常见根因。
2. 用户从早期 adopter 扩展到普通开发者，shell form 的隐性依赖
   （基础镜像的 sh 差异：bash vs dash vs busybox）变成了可移植性问题。

## 重构 — 如果重来

本质需求：在构建时执行命令，命令间共享上下文。

替代路径：
- A：exec form 作为默认，shell 特性通过显式 opt-in（如 `RUN --shell`）。
  默认安全，需要 shell 的用户主动声明。
- B：Docker 内部实现跨平台命令包装器，不依赖基础镜像的 shell。
  实现复杂度高，兼容性风险大。
- C：保持现状，构建时检测不当使用并 warn。

## 对比 — 新方案更好吗

推荐 A。替换成本低（默认值变更 + 文档），消除"信号不转发"高频痛点。
B 风险太大——数百万 Dockerfile 依赖 sh 行为。

> **最终判断**：值得替换。和 Step 1 的"前提已变"定性一致——如果 Docker 还是 2013 年的定位，结论会不同。
```

## 注意事项

- **Step 1 和 Step 3 必须自洽**：Step 3 的结论必须能回溯到 Step 1 的定性。不自洽是框架被架空的最大信号。
- **L1 不是偷懒**：一句话判断背后走了三步思考，是压缩后的洞察。
- **不知道就问，不要猜**：L2/L3 中如果缺少关键上下文，停止并提问。给出一个信息不足的"报告"比直接说"我需要更多信息"更不负责。
- **假设显式声明**：基于假设得出的结论，把假设说清。让用户知道哪条假设变了结论也会变。
- **具体 > 抽象**：用具体的代码、数据、场景支撑论点。
