# knowledge-summary — 深度理解概念

帮助用户真正理解一个概念。核心不是给定义，而是帮用户走过三段认知递进：

1. **"它像什么？"** — 用类比建立直觉（一旦生成就不再修改）
2. **"但它不完全是……"** — 明确指出类比的局限，修正认知偏差
3. **"它到底是什么？"** — 给出精确定义和专业拆解

简单来说：先让用户觉得"哦我懂了"，再告诉用户"等等那里不对"，最后让用户"现在我真的懂了"。

## 什么时候用

**应该触发：**
- 用户请求理解陌生概念（"帮我理解 X"、"解释 X 是什么"、"X 的原理是什么"）
- 用户请求用类比讲解（"用大白话讲"、"打个比方"）
- 用户阅读了文档/论文后需要提炼核心知识
- 用户对已有理解有疑问需要纠正偏差
- 用户想扩展某个概念的知识边界

**不应该触发：**
- 简单翻译、普通文本摘要、代码调试 — 除非用户明确说要"深入理解"


## 和用户沟通的方式

这个 skill 的核心价值是帮用户**建立认知模型**，不是给一个标准答案。所以：

- 尽量用类比和故事，而不是罗列事实。三段式的本质是"先让人听懂，再让人修正，最后让人精确"。
- 对用户的理解进行评分 (0-10) 时给出理由，让用户知道哪里对、哪里偏了。
- 回答是根据用户的问题和自身理解，基于 `knowledge.md` 寻找相关内容，但回答不是将 `knowledge.md` 全部输出给用户，而是提炼出与问题相关的部分。
- 如果用户提供的描述有严重偏差，温和地指出来，不要只说"不对"。
- 如果用户只是丢过来一个概念名，没有文档，可以先问"你有相关的文档或资料吗？我们一起学习"。没有文档就靠已有知识或搜索时，要明确告知用户。
- 如果用户给出的文档长度过长，需要向用户表明：因为能力有限会无法一次性完整读完，在单次阅读完成后，询问用户是否要继续阅读完整文档。

## 工作方式

本 skill 围绕一个工作台目录运行。在当前工作目录下：

```
workbench/
├── answer.md              # 三段式回答文档
├── questions.md           # 记录所有提出的问题
└── documents/
    ├── knowledge.md       # 从原始文档提炼的知识库（核心）
    └── <原始文档>         # 待学习的参考文档
```

## 三种工作模式

### 模式 1：初始化 — 理解一个新概念

当用户没有给出任何问题或者描述信息时，执行这个模式 1。

从头理解一个概念。**所有回答必须基于真实文档或 knowledge.md 的知识，禁止编造。**

为什么要走这套流程？因为直接从文档读出的知识，写进三段式时才有根有据。跳过读文档直接写类比，容易产生似是而非的内容。

1. **扫描并学习文档** — Glob 扫描 `documents/*`，对每个文档执行预览→速读→精读三遍阅读法（模板见下方），提取与概念直接相关的核心内容。
2. **写入 knowledge.md** — 用自己的话重述，按章节组织。已有文件则追加新章节，不覆盖已有内容。
3. **生成 answer.md** — 基于 knowledge.md 生成三段式文档。注意："它像什么？"被确定后，**后续流程不可再修改**。这是因为类比建立的是第一印象，频繁更换类比会让用户困惑。
4. **呈现给用户** — 输出 answer.md 全部内容，提示可以用更多问题或文档深入。

### 模式 2：问答 — 修正认知偏差

用户读了 answer.md 后产生疑问，或者想验证自己的理解。

1. 读取 `documents/knowledge.md`，不存在则提示先初始化。
2. 搜索 knowledge.md 中与问题相关的章节。
3. 在 answer.md 的"但它不完全是……"章节新增认知偏差条目（`| 你可能会觉得… | 实际上的精确真相是… |`）。
4. 如果用户描述中包含自己的理解，**对理解进行评分 (0-10)** 并给出理由。
5. 将问题追加到 `questions.md`。

为什么偏差条目要追加而不是覆盖？因为用户的认知是逐步深入的，保留历史偏差能让用户看到自己的进步轨迹。

### 模式 3：补全 — 扩展知识边界

用户引用了新文档要求学习，用户给出了 documents/<path>/<file> 的格式时，执行模式 3。

1. 验证文档路径是否存在，不存在则报错。
2. 精读文档（预览→速读→精读）。
3. 在 knowledge.md 中追加新章节，标注当前日期和来源路径。
4. 在 answer.md 的"它到底是什么？"章节中增强或新增条目。"它像什么？"和"但它不完全是……"保持不变。
5. 输出更新后的"它到底是什么？"。
6. 这里给出内容不应该存在类别的描述，而更多应该是陈述。

为什么只更新第三段？因为新文档补充的是专业知识，不改变已有的类比框架和偏差修正。

## 模板

### answer.md 模板

```markdown
# 理解 `[核心关键词]`

## 一句话抓住本质
[用一句话讲清它解决什么问题，或者它的独特价值]

## 第一步："它像什么？"

### 类比场景
[用故事或场景描述]

### 对应关系
- **[场景元素 A]** ⇔ [概念中的 X]
- **[场景元素 B]** ⇔ [概念中的 Y]

## 第二步："但它不完全是……"

| 你可能会觉得… | 实际上的精确真相是… |
| :--- | :--- |
| [修正的错误环节] | [正确的解释] |

> **一句话**：让你看到 [核心逻辑]，本质上是 [一句话说清]。

## 第三步："它到底是什么？"

### 专业定义
> [标准定义，引用来源]

### 人话拆解
- **"[术语]"** — [通俗解释]

### 为什么要有这个概念？
[设计动机、解决什么问题]

## 常见误解快查（可选）
- ❌ "误解" → ✅ "真相"
```

### 阅读方法（预览→速读→精读）

1. **预览**: 读标题、摘要、小标题、结论、图表标题。确定知识领域和章节骨架。不要读方法/实验细节。
2. **速读**: 根据骨架读每段第一句和最后一句。总结为"这些段落讲了什么？这能回答什么问题？"
3. **精读**: 只对难懂或关键部分深入。读完后做到"标题 + 一句话核心贡献 + 两个最强证据 + 一个批评"。
4. 结束时用三句话复述文档的论证脉络。

### knowledge.md 格式

```markdown
# executor's perception

## 元信息
- 已学习文档列表：
  - [x] `man.md:1;157 lines`
- 知识领域：[领域名]

---

## `文档路径`

### 小标题

> **速读总结**（1-3句）
> **可回答问题**：

#### 精读笔记
- **关键概念**：概念名（`Lxx-Lyy`）：解释
- **证据/数据**：`Lxx-Lyy`：引用原文
- **质疑/批评**：`Lxx-Lyy`：疑问或局限
- **关联**：与其他内容的关系

### 三句话复述
1. 问题：解决了什么？
2. 方案：如何解决？
3. 结果：达成什么？
```

完整的 answer.md 和 knowledge.md 实例见 `example/` 目录（包含真实的 TCP 三次握手案例）。

## 关键约束

| 项目 | 规则 |
| :--- | :--- |
| 知识来源 | 优先从 `knowledge.md` 读取，不存在时或者内容为空时才读原始文档|
| 超长文档 | 先阅读前面的 3000 lines，然后需要在 `knowledge.md` 的 元信息注明已经读的数，分阶段补全 `knowledge.md` |
| 可读写范围 | 读 `documents/*`，写 `answer.md`、`questions.md`、`documents/knowledge.md` |
| "它像什么？" | 初始化后不可修改 |
| "但它不完全是……" | 问答模式下更新，含评分 |
| "它到底是什么？" | 补全模式下增强 |
| questions.md | 每次交互均追加，不覆盖 |
| knowledge.md | 每次编辑完后都需要检查当中的元信息，检查已读文档的 lines 是否和 `wc -l  documents/<file>` 的结果相差过大|
