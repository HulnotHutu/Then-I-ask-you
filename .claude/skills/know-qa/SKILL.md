# know-qa — 认知修正

核心任务：用户对已有的知识表达了自己的理解或提出了新疑问，基于 `knowledge.md` 在 `answer.md` 中追加修正条目，帮助用户发现"哪里对"和"哪里偏了"。

修正条目用**追加而非覆盖**的方式记录——保留历史偏差条目能形成"认知修正链"，让用户回头看到自己的理解演变。

## 前提

- `documents/knowledge.md` 和 `answer.md` 都必须存在
- 如果不存在，提示用户知识库或认知文档尚未就绪，需要先提供文档进行学习
- 如果用户引用了新文档路径要求学习文档，不属于本技能的处理范围

## 工作流程

### 1. 理解用户输入

用户输入必定包含至少一个：

- **问题**：关于概念的追问（"为什么 X 会 Y？"、"那如果 Z 呢？"）
- **描述**：用户自己对该概念的理解描述（"你看我这样理解对不对：X 就是……"）

**从用户输入中提炼核心问题/误解**——用 1-2 句话精炼用户真正想表达的疑问或认知偏差是什么。这是后续写入修正条目左列的基础。

### 2. 搜索 knowledge.md 相关内容

读取 `documents/knowledge.md`，定位与用户问题**最相关的章节**：

- 优先按关键词匹配
- 如果用户的问题涉及多章节内容，优先引用描述最详细的章节
- 不要重新扫描原始文档——只从 knowledge.md 中查找

### 3. 追加修正条目到 answer.md

在 `answer.md` 的"但它不完全是……"表格中**追加新行**：

```
| [从用户输入提炼的误解/疑问] | [基于 knowledge.md 的正确解释] |
```

**左列规则**（从用户输入提炼）：
- 如果用户给了自己的理解描述 → 提炼出其理解中与 knowledge.md 有偏差的核心观点
- 如果用户只提了问题 → 提炼出一个"一般人遇到这个问题时容易产生的误解"
- 必须是具体的观点，不是泛泛的"对 X 理解不对"

**右列规则**（基于 knowledge.md）：
- 必须从 knowledge.md 中找到对应的精确描述作为依据
- 标注来源章节（如 `tcp.txt §3.4`）：让用户能追溯

### 4. 评估用户的理解（仅当用户给出了自己的描述时）

如果用户输入中包含自己的理解描述，从以下三个维度评估：

#### 核心概念：哪里对、哪里偏
- 指出描述中与 knowledge.md 吻合的部分（肯定的部分）
- 指出描述中与 knowledge.md 偏离的部分
- 用具体的"这里对/这里偏"让用户知道该保留什么、该调整什么

#### 边界条件：这句话生效的范围
- 用户的理解在什么条件下成立？在什么条件下不成立？
- 从 knowledge.md 中找到相关的限制条件或适用范围
- 帮助用户建立"这个概念不是万能的，它有适用边界"的意识

#### 术语使用：是否使用了引人误解的词语
- 指出用户描述中可能引起歧义的术语
- 如果用户用了一个"日常词"来描述一个"专业概念"，明确指出这两者之间的差异
- 给出更精确的术语替代建议

评估输出格式：
```
**核心概念**：吻合 — [具体吻合的部分]；偏离 — [具体偏离的部分]
**边界条件**：[用户理解成立的场景] vs [knowledge.md 中更精确的限定]
**术语使用**：[可能引起误解的用语] → 建议使用 [更精确的术语]
```

### 5. 追加 questions.md

在 `questions.md` 中追加本次问答记录：

```markdown
### YYYY-MM-DD HH:MM
- **问题**：[用户提出的问题 / 描述的核心]
- **回答要点**：[本次回答的核心要点，1-3 条]
```

### 6. 输出

向用户展示：
1. 本次追加的修正条目（表格行）
2. 评估结果（如有）——核心概念、边界条件、术语使用三个维度
3. 更新的 questions.md 记录

## 约束

| 项目 | 规则 | 原因 |
| :--- | :--- | :--- |
| 知识来源 | 只从 `knowledge.md` 读取，不重新扫描原始文档 | knowledge.md 已经是提炼后的知识 |
| 权限 | 追加写 `answer.md`（仅"但它不完全是……"表格）和 `questions.md`，不修改 `knowledge.md` | 问答影响呈现层和记录，不影响知识库 |
| 修改范围 | 只追加到"但它不完全是……"表格行，不动"它像什么？"和"它到底是什么？" | 类比框架和精确定义在生成时已确立，不能随意改动 |
| 修正条目来源 | 不直接使用用户的原话写左列，需提炼为具体的误解/疑问 | 用户可能表达冗长，提炼后更聚焦 |
| questions.md | 每次追加新条目，使用时间戳格式 | 完整记录对话脉络，方便回溯 |
