# 理解 `TCP 三次握手`

## 一句话抓住本质

**三次握手**是 TCP 为了在不可靠网络中**同步双方初始序列号**并协商连接参数而设计的三步消息交换，防止历史连接请求的干扰。

## 第一步：“它像什么？”

### 类比场景

你想跟一位从未见过面的远距离朋友建立暗号通信，而且你们之间用的信使可能会把旧信乱塞进来。  

- 你第一次写信：“我接下来用编号 100 开始编号，你收到了吗？”  
- 朋友回信：“我收到你的 100，我用 300 开始编号，你收到了吗？”  
- 你再回信：“收到你的 300，确认。”  
此时你们确认了彼此的起始编号，对话正式开始。

### 对应关系

- **写信方** ⇔ TCP 主动打开端（客户端）
- **收信方** ⇔ TCP 被动打开端（服务器）
- **第一封信中的“编号100”** ⇔ 初始序列号（ISS）
- **朋友回信中的“收到100”** ⇔ 确认号（ACK = ISS+1）
- **朋友回信中的“我用300开始”** ⇔ 自己的 SYN（ISN = 300）
- **最后一封“确认300”** ⇔ 对 SYN 的确认（ACK = IRS+1）

## 第二步：“但它不完全是……”

| 类比中你可能会以为… | 实际上的精确真相是… |
| :--- | :--- |
| 三次握手是为了“确认双方都在线” | 它的**首要目的**是同步初始序列号（ISN），顺便才验证对方在线。没有序列号同步，就无法区分新旧数据。 |
| 三次握手必须严格三步 | 第二步（SYN+ACK）可以合并，但实际上也可以拆成两步（SYN 和 ACK 分开），但标准实现为一步。 |
| 旧信件会被自动丢弃 | TCP 通过“时钟化 ISN + 三次握手 + RST”机制**主动发现并拒绝**旧重复的 SYN。若没有三次握手，旧 SYN 可能导致错误建立连接。 |
| 三次握手完成后才能发数据 | 数据可以**携带在第三次握手的 ACK 段中**（如 HTTP 的 TFO 优化），但接收方在连接 ESTABLISHED 前不会将数据交给应用。 |

> **一句话**：让你看到**三次握手本质是序列号同步协议**，本质上是**一种带确认的、双向状态同步的分布式共识过程**，而不是简单的“打招呼”。

## 第三步：“它到底是什么？”

### 专业定义
>
> “The three-way handshake reduces the possibility of false connections. It is the implementation of a trade-off between memory and messages to provide information for this checking.” — RFC 793 Page 30–31

**人话拆解**：

- **“SYN（同步）”** — 就是一方说“我打算从此编号开始发送数据，请确认”。
- **“ACK（确认）”** — 回应“我已收到你的同步请求，接下来请发序号+1的数据”。
- **“初始序列号（ISN）”** — 每个连接独立生成的 32 位起始序号，由时钟驱动（约 4μs 递增）。

### 为什么要有这个概念？

因为网络会延迟、复制或重排包。如果没有三次握手，一个**延迟很久的老 SYN** 可能在新连接建立后到达，导致接收方误以为是新请求。通过三次握手和时钟化 ISN，接收方可以要求发送方“确认这个 SYN 是不是我最近发出的”，从而拒绝旧请求。

## 常见误解快查

- ❌ “三次握手就是为了建立连接，跟安全无关” → ✅ 它核心是为了同步序号，间接防止历史连接的报文干扰。
- ❌ “收到 SYN 后直接建立连接就完事了” → ✅ 必须回复 SYN+ACK 并等待第三次 ACK，否则无法确认对方收到了自己的 SYN。
- ❌ “两轮消息就够了（SYN + ACK）” → ✅ 两轮只能让主动方知道被动方在线，但被动方无法确认主动方收到了它的 SYN，第三次 ACK 是必要的。
