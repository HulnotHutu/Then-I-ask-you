# 理解 TCP SACK（选择性确认）
>
> D-SACK（重复选择性确认）、累积确认、拥塞控制

## 一句话抓住本质

TCP SACK 让接收方告诉发送方"我具体收到了哪些碎片数据"，发送方只重传真正丢失的部分——而不是像原来那样每个往返只能猜一个。

## 第一步："它像什么？"

### 类比场景

想象你是一位快递员，负责把一整套百科全书（100 卷）送到客户手中。但快递公司不靠谱，包裹经常在路上弄丢。

**没有 SACK 的世界（累积确认）：**
客户收到书后，只能给你寄一张明信片说："我已经收到了第 1 卷到第 X 卷。" 如果第 5 卷和第 8 卷丢了，但第 6、7 卷已经到了，客户只能告诉你"我收到了第 1-4 卷"。你完全不知道第 6、7 卷其实已经到了，只能一个一个地猜哪个丢了——先重发第 5 卷，等客户的回信，再决定下一步。效率极低。

**有 SACK 的世界：**
客户收到书后，在明信片上写："我收到了第 1-4 卷，另外第 6、7 卷也到了。" 你一看就知道：只需要补第 5 卷和第 8 卷。一次性搞定。

**有 D-SACK 的世界（更进一步）：**
有时候你担心包裹丢了，提前又发了一次第 5 卷。客户收到后告诉你："第 5 卷我之前已经收到了，你又发了一遍。" 你意识到——哦，原来不是丢了，是网络延迟，没必要慌张地缩减发货量。

### 对应关系

- **百科全书** ⇔ TCP 要传输的数据流
- **快递员** ⇔ TCP 发送方
- **客户** ⇔ TCP 接收方
- **明信片（ACK）** ⇔ TCP 确认报文
- **"我收到了第 1-4 卷"** ⇔ 累积确认号（Cumulative ACK）
- **"另外第 6、7 卷也到了"** ⇔ SACK 块（报告非连续已收数据）
- **"第 5 卷我之前已经收到了"** ⇔ D-SACK 块（报告重复收到的数据）
- **TCP 选项空间** ⇔ 明信片背面有限的空白区域（只能写几个条）

## 第二步："但它不完全是……"

| 你可能会觉得… | 实际上的精确真相是… |
| :--- | :--- |
| SACK 让接收方"确认"了某块数据，发送方就可以放心了 | SACK 是**建议性（Advisory）**的——接收方报告"收到了"之后仍然可以丢弃该数据（称为 reneging）。发送方必须等累积 ACK 真正推进窗口左沿，才能释放缓冲区。 |
| 第一个 SACK 块只是随便挑一个已收到的块报告 | 第一个 SACK 块**必须包含触发这个 ACK 的段所在的数据块**（除非该段推进了累积确认号）。它反映的是接收方缓冲区"最新的变化"，而不是"最大的"或"最左边的"。 |
| D-SACK 需要双方额外协商才能使用 | D-SACK **不需要单独的协商握手**。只要连接协商了 SACK 能力（SYN 中交换 SACK-permitted），D-SACK 自动可用。不支持 D-SACK 的发送方会像没看见一样忽略 D-SACK 块。 |
| 发送方收到 SACK 信息后可以立刻释放已确认数据的内存 | 发送方**不能只根据 SACK 释放内存**。段必须等到累积 ACK 推进窗口左沿后才能出队释放。SACK 只影响"要不要重传"，不影响"能不能释放"。 |
| D-SACK 报告重复段时，和普通 SACK 块各占各的位置 | D-SACK **占用的是第一个 SACK 块的位置**。也就是说，一份 ACK 包里，第一个块是 D-SACK（重复段信息），从第二个块开始才是 RFC 2018 定义的普通 SACK 块。 |

| SACK 是"提高 ACK 流程的速度"，让确认跑得更快 | SACK **不改变 ACK 的发送频率或速度**。接收方该什么时候发 ACK 还是什么时候发。SACK 改变的是每个 ACK **携带的信息密度**——传统 ACK 只告诉你"我连续收到哪了"，SACK 额外告诉你"我碎片式地收到了哪些"。不是把明信片寄得更快，而是在同一张明信片上写更多内容。 |
| 有了快速重传就不需要超时重传了 | 两者是**互补而非替代**关系。快速重传靠收到 3 个重复 ACK 来"猜"丢包（约 1 RTT 就能检测），但重复 ACK 可能被全部丢失或数据全部在途中（零窗口）。超时重传是**最后的兜底机制**——不管什么情况，只要定时器到期就重传。但代价极大：RTO 后拥塞窗口重置为 1，进入慢启动。 |
| SACK 改变了 TCP 拥塞控制的基本规则 | SACK **保留**了拥塞控制的全部核心机制（慢启动、拥塞避免、快恢复）。RFC 2018 明确说"de facto standard TCP implementations MUST be preserved"。SACK 只影响"我该重传哪些段"，不影响"我该以多快的速度发送"（拥塞控制决定后者）。 |
| 滑动窗口和拥塞窗口是同一个东西 | 它们是**两层独立的窗口**，但最终生效的是两者的最小值。**滑动窗口（Advertised Window）** 由接收方通告，反映接收端还剩多少缓冲区——这是流控，防止"发送方淹死接收方"。**拥塞窗口（Congestion Window, cwnd）** 由发送方自己维护，反映网络的承载能力——这是拥塞控制，防止"发送方淹死网络"。两者共同决定实际可发送的数据量：`可发送字节数 = min(cwnd, rwnd)`。 |
| 滑动窗口就是"接收方告诉发送方最多发多少数据" | 这不算错，但漏掉了一个关键维度——滑动窗口不仅是"容量上限"，更是**确认驱动**的。窗口左沿只在收到累积 ACK 后才向右滑动。换句话说：不是"我允许你发 N 个字节"就完了，而是"你发完 N 个字节后必须等我的 ACK 才能再发下一批"。这种"发一批 → 确认 → 再发一批"的模式就像一个不断向前滚动的窗口，所以叫"滑动"窗口。 |
| 接收方通告的窗口大小（rwnd）是实时精确的 | 接收方通告的窗口只是一个**快照**——它反映的是接收方在发出那个 ACK 时刻的缓冲区余量。但发送方可能随后就发出数据，接收方的缓冲区会继续缩小。极端情况下（接收方处理慢、ACK 在路上），rwnd 到达发送方时已经"过时"了。这就是为什么 TCP 有**零窗口探测**机制——发送方必须定期探测，防止接收方缓冲区已释放但窗口更新 ACK 丢失导致死锁。 |
| 接收窗口 rcv_wnd 是一个固定值，连接建立后就不再变化 | `rcv_wnd` 是**动态计算**的——它是接收缓冲区的实时剩余空间。每当上层应用 recv 取走数据、或新的段到达填满缓冲区，`rcv_wnd` 都会变化。此外它还被 `window_clamp` 封顶（可通过 `SO_RCVBUF` 设置），不能无限扩大。通告窗口 = min(实际可用缓冲区, window_clamp)。 |
| 预读队列（prequeue）只是接收队列的另一个名字 | prequeue 和 receive_queue 是**两条完全不同的数据路径**。正常路径：段到达 → 通过累积 ACK 检查 → 放入 `sk_receive_queue` → 应用 recv 时从队列拷贝。预读路径：应用已在 recv 中阻塞等待 → 段到达时**直接拷贝到用户空间**（`ucopy.iov`），完全绕过 `sk_receive_queue`。这是一种性能优化（VJ 风格），避免了一次内核到内核的拷贝和上下文切换。 |
| 所有落在接收窗口内的乱序数据都会进 `out_of_order_queue` | 进入乱序队列需**同时满足四个条件**：(1) 通过 RFC 793 序列号可接受性测试——`SEG.SEQ` 必须在 `RCV.NXT` 到 `RCV.NXT+RCV.WND-1` 之间，或其尾部落在窗口内；(2) `SEG.SEQ > RCV.NXT`（有缺口——如果 SEQ 正好等于 `RCV.NXT`，数据是顺序的，直接进 `receive_queue` 或走 `prequeue` 快速路径）；(3) `SEG.LEN > 0`（携带实际数据，纯 ACK 不进数据队列）；(4) 数据不是完全重复（至少部分字节是首次收到，完全重复的段只回 ACK，数据丢弃）。**零窗口时不接受任何携带数据的段**（`RCV.WND=0 && SEG.LEN>0` → not acceptable），但 RST/ACK/URG 仍会被处理。 |
| 乱序队列里的数据需要等待发送方重传才能用 | 乱序队列保存的是**已经成功接收的数据**，只是中间有缺口。一旦缺失的段（`RCV.NXT` 位置的段）到达，内核会将 `out_of_order_queue` 中与 `RCV.NXT` **连续**的所有段一次性转移到 `sk_receive_queue`，并推进 `RCV.NXT`。这些数据不会丢失，也不依赖重传——它们已经"到了"，只是在等待前面的缺口被补上。SACK 的存在就是为了让发送方知道"这段数据已经在乱序队列里了，不用重传"。 |
| 拥塞控制是 RFC 793 的一部分 | RFC 793 **完全没有定义拥塞控制**——它只定义了重传超时（RTO）作为可靠性的兜底机制。TCP 拥塞控制直到 1988 年 Van Jacobson 的论文才被引入（观察到当时互联网多次发生"拥塞崩溃"后提出的补救方案）。`linux-TCB.md` 中看到的 `snd_cwnd`、`snd_ssthresh`、`ca_state` 等字段都是后来添加到 TCB 中的。RFC 793 的 RTO 只有一阶平滑（`RTO = β × SRTT`），Jacobson 1988 才引入了 RTT 方差项（RTTVAR）。 |
| 发送方有个窗口就够了——既然有接收方通告的 rwnd，为什么还要 cwnd？ | 两个窗口保护**不同的对象**：rwnd（接收方通告窗口）保护接收方缓冲区不被撑爆——这是**端到端**约束。cwnd（拥塞窗口）保护**网络**不被撑爆——中间路由器的队列是共享资源，发送方看不到接收方的 rwnd 也看不到路由器的状态。可发送字节数 = `min(cwnd, rwnd)`，两者取小。这个"两窗取小"的设计是 TCP 最优雅的部分之一——发送方在不了解网络内部状态的情况下，通过丢包信号（或 ECN）来猜测网络的承载能力。

> **一句话**：让你看到 TCP 如何精打细算地利用 ACK 选项区有限的几十个字节，本质上是**用"告诉对方确切少了什么"替代"让对方猜少了什么"**；而滑动窗口则是 TCP 的"水龙头"——接收方随时调整口径，确保水流（数据）不会溢出水桶（缓冲区）。

## 第三步："它到底是什么？"

### 专业定义
>
> TCP 选择性确认（SACK）是一种 TCP 选项扩展机制，允许数据接收方在 TCP 确认报文段中报告一个或多个非连续已接收数据块的边界（左右序列号），使发送方能够精确推断丢失的段并只进行选择性重传。SACK 由 RFC 2018 标准化，D-SACK 扩展（RFC 2883）进一步允许接收方报告重复接收的段。

> 所有 SACK 机制建立在 RFC 793 定义的 TCP 基础框架之上：**面向连接、字节流序号、累积确认、超时重传、滑动窗口流量控制**。SACK 不改变这些基础，而是在"累积确认"的盲区——非连续已收数据——上增加了信息通道。

### 人话拆解

- **"SACK-permitted（Kind=4）"** — 就是连接建立前（SYN 阶段）双方打个招呼："我支持 SACK"，后续通信中才能使用 SACK 选项。
- **"SACK 块（Block）"** — 一个 [Left Edge, Right Edge) 的区间。Left Edge 是块内第一个字节的序号，Right Edge 是块外第一个字节的序号。每个块 8 字节。
- **"Advisory SACK / Reneging"** — 接收方可以反悔：报告了"我收到了 X"，但后续因缓冲区不足可以丢弃 X。因此 SACK 只影响"重传决策"，不影响"内存释放"。
- **"累积 ACK"** — TCP 头部那个 4 字节的 Acknowledgment Number，表示"我连续收到了直到序号 N 的所有数据"。是唯一的正式确认依据。
- **"D-SACK"** — 第一个 SACK 块报告的内容如果是重复数据（序号空间小于同包累积 ACK），就是 D-SACK。告诉发送方"你发的东西我这里已经有过一份了"。
- **"SND.UNA / SND.NXT / SND.WND"** — RFC 793 定义的核心发送变量。SND.UNA = 最老的未确认序号，SND.NXT = 下一个要发的序号，SND.WND = 发送窗口大小。这三个变量划定了发送序列空间的三个区域：已确认的旧数据、已发未确认、可发新数据。SACK 选项的到来会影响"哪些段可以跳过重传"，但**不会改变这三个变量推进的规则**——只有累积 ACK 才能推进 SND.UNA。
- **"RTO（重传超时）"** — RFC 793 基于 SRTT（平滑 RTT）计算：RTO = β × SRTT（β ≈ 1.3~2.0），有上下界。发送方将数据放入重传队列并启动定时器——"若定时器到期前 ACK 未到，则重传"。这是 TCP 可靠传输的**最终兜底手段**。
- **"RST（重置）"** — 处理异常连接的"核按钮"。当收到明显不属于当前连接的段时（如序列号不在窗口内、安全级别不匹配），发送 RST 强制对方关闭连接。三次握手中的旧重复 SYN 就是通过 RST 来消除的。
- **"ISN（初始序列号）"** — 基于 32 位时钟选择，每 4 微秒递增一次，循环周期约 4.55 小时（远大于 MSL=2 分钟），确保不同连接实例的序列号不会碰撞。
- **"段到达八步处理"** — RFC 793 3.9 节定义的 SEGMENT ARRIVES 事件按严格顺序处理：序列号检查 → RST → 安全/优先级 → SYN（已建连后再收到 SYN 是错误） → ACK（推进 SND.UNA、删除重传队列中已确认段、更新窗口） → URG → 段文本（推进 RCV.NXT、调整 RCV.WND、发送 ACK） → FIN。每一步都可能因为合法性检查失败而提前返回或发送 RST。
- **"重传超时处理"** — 3.9 节定义得非常简短：重传重传队列前端的段，重新初始化定时器。后来 Jacobson 1988 等拥塞控制算法在此基础上大幅增强了 RTO 退避、慢启动等机制。
- **"窗口更新保护机制（SND.WL1/SND.WL2）"** — 防止用旧的、延迟到达的段来更新发送窗口。更新条件：`SND.WL1 < SEG.SEQ` 或 `(SND.WL1 == SEG.SEQ && SND.WL2 <= SEG.ACK)`。保证了只有包含最新信息的段才能修改窗口。

### 为什么要有这个概念？

没有 SACK 时，TCP 在多丢包场景下会遭遇灾难性吞吐下降——每个 RTT 只能发现一个丢包。SACK 打破了这种"一次猜一个"的困局，让 TCP 在丢包率高的网络（如无线网络、卫星链路）中仍然保持可用的吞吐量。D-SACK 则更进一步，让 TCP 能区分"真的丢包"和"假丢包（重排序、复制、过早超时）"，避免因错误判断而过度收缩拥塞窗口。

但更重要的是理解 SACK 在 TCP 协议栈中的**定位层次**：RFC 793 定义了 TCP 的"硬件"（头部结构、状态机、确认机制），而 SACK（RFC 2018）和 D-SACK（RFC 2883）是在这个"硬件"上添加的"软件扩展"——就像给一台汽车加了更好的导航系统，没有改变发动机和底盘。

### 什么是滑动窗口（Sliding Window）？

**一句话**：滑动窗口是 TCP 的**流量控制机制**。接收方在每次 ACK 中通告自己的剩余接收缓冲区大小（Window 字段），发送方据此限制"已发但未确认"的数据量，保证发送速率不超出接收方的处理能力。

**核心机制**（人话版）：

- **窗口左沿（SND.UNA）**：最老的未被确认的字节序号。只有累积 ACK 才能推动左沿向右。左沿不动，发送方不能释放对应的发送缓冲区。
- **窗口右沿（SND.UNA + SND.WND）**：允许发送的最后一个字节的序号。这个值由接收方每次 ACK 时更新通告的 `Window` 字段决定。
- **可发送的新数据**：在 `SND.NXT`（下一个要发的字节）和右沿之间的空间。如果 SND.NXT 碰到了右沿，发送方必须停止发送，等待接收方通告更大的窗口。

**为什么叫"滑动"？** 因为窗口的边界是动态的——收到 ACK 后左沿向右滑（释放缓冲区），接收方缓冲区腾出空间后右沿也向右滑（扩大窗口）。整个窗口像一个在序列号空间上不断向右移动的滑块。

**滑动窗口和 SACK 的关系**：滑动窗口决定"我能发多少"（流控），SACK 决定"丢了该补发哪个"（精确重传）。两者互补——滑动窗口是基础层的流控机制（RFC 793 就已定义），SACK 是选项层的精确重传增强（RFC 2018）。没有滑动窗口，SACK 无从谈起；没有 SACK，滑动窗口在丢包时效率低下。

### 什么是 TCB（传输控制块）？

**一句话**：TCB 是 TCP 为每个连接维护的"档案袋"——RFC 793 定义了它需要记录什么（序列号、窗口、定时器、状态机），Linux 在 `struct tcp_sock` 中具体实现了它。

**从 RFC 到 C 代码的映射**：

| RFC 793 概念 | Linux 2.6.12 实现 | 含义 |
|:---|:---|:---|
| RCV.NXT | `tcp_sock.rcv_nxt` | 期望接收的下一个字节 |
| SND.NXT | `tcp_sock.snd_nxt` | 下一个要发送的字节 |
| SND.UNA | `tcp_sock.snd_una` | 最老的未被确认字节 |
| SND.WND | `tcp_sock.snd_wnd` | 接收方通告的窗口 |
| RCV.WND | `tcp_sock.rcv_wnd` | 本地接收窗口 |

**TCB 内部的 5 个缓冲区**（Linux 实现的特有细节）：

| 缓冲区 | 存放内容 | TCP 状态视角 |
|:---|:---|:---|
| `sk_write_queue` | 上层已提交但未被 ACK 的数据 | SND.NXT ~ SND.UNA+SND.WND |
| `sk_receive_queue` | 已 ACK 顺序到达但上层未取走 | RCV.NXT 之前 |
| `out_of_order_queue` | 乱序到达的数据段 | RCV.NXT 之后，中间有缺口 |
| `sk_error_queue` | ICMP 错误（如端口不可达） | 异步事件，等待上层读取 |
| `prequeue` (ucopy) | 预读缓冲，绕过 receive_queue | 性能优化（VJ 风格） |

**关键理解**：`sk_write_queue` 同时承担了**三个角色**——发送队列（存上层交下来的数据）+ 重传队列（超时后从中取出重发）+ 待确认队列（ACK 到达后从中删除）。Linux 不需要单独的重传队列，因为"已发送但未确认"的段自然就在发送队列中，重传时扫描该队列找到超时的段即可。

**ICMP 错误如何找到"主人"？** 当一个路由器无法转发 TCP 包时，会生成 ICMP 差错报文。RFC 792 规定 ICMP 报文必须携带原始 IP 包载荷的前 8 字节——对 TCP 来说，这 8 字节正好是源端口和目的端口。内核用 ICMP 中的 IP + TCP 首部的端口拼出完整四元组，在 socket 哈希表中精确找到对应连接，将错误码存入 `sk_error_queue`。

**TIME_WAIT 的内存优化**：TIME_WAIT 状态的连接不再收发数据，不需要完整的 `tcp_sock`。Linux 用轻量级 `struct tcp_tw_bucket` 替代，只保留序列号、窗口、超时时间等必需字段。在繁忙的 Web 服务器上，这能节省大量内存。

**数据结构层次**：
```
struct sock           ← 所有协议的通用套接字
  └── struct inet_sock ← IPv4 特定（IP、端口）
      └── struct tcp_sock ← TCP TCB（序列号、窗口、拥塞等）
```
这是 Linux 内核经典的分层继承模式——通用代码操作 `sock`，TCP 代码操作 `tcp_sock`。

**RFC 793 TCB vs Linux tcp_sock 的关键差异**：

| 方面 | RFC 793 TCB | Linux tcp_sock |
|:---|:---|:---|
| 拥塞控制 | 未定义（仅为重传超时） | 完整字段：cwnd, ssthresh, packets_out, lost_out, sacked_out |
| SACK | 未定义（SACK 在 RFC 2018） | `duplicate_sack[1]` + `selective_acks[4]` |
| 拥塞算法 | — | 可插拔：Vegas, Westwood, BIC 各有独立结构体 |
| RTT 测量 | SRTT + RTO（一阶平滑） | srtt(<<3 定点数) + mdev + rttvar（Jacobson 1988 改进） |
| TIME_WAIT | 标准 11 状态之一 | 独立轻量结构 `tcp_tw_bucket` |

### TCB 接收窗口的限定参数

接收窗口 `rcv_wnd` 并不是一个独立存在的变量——它由以下参数共同决定：

```
通告窗口 = clamp(实际可用缓冲区空间, 0, window_clamp)
```

其中：

| 参数 | 结构体字段 | 作用 |
|:---|:---|:---|
| **接收缓冲区容量** (`sk_rcvbuf`) | `sock.sk_rcvbuf` | 套接字接收缓冲区的总大小，可通过 `SO_RCVBUF` 设置。内核通常会将其翻倍（用于容纳元数据开销） |
| **已占用的缓冲区空间** | — | 当前遗留在 `sk_receive_queue` + `out_of_order_queue` 中的数据量。应用 recv 取走数据后，这部分空间被释放 |
| **实际可用空间** | — | `sk_rcvbuf - 已占用空间`，即剩余缓冲区字节数 |
| **窗口上限** (`window_clamp`) | `tcp_sock.window_clamp` | 窗口的硬上限，在连接建立时根据 `sk_rcvbuf` 和 MSS 计算得出。即使缓冲区完全空了，通告的窗口也不会超过这个值 |
| **最大观察窗口** (`max_window`) | `tcp_sock.max_window` | 记录从对端看到的最大通告窗口，用于发送侧判断对方缓冲区是否在收缩 |

**核心逻辑**（基于 Linux 内核接收缓冲区管理的通用机制）：

1. **计算可用空间**：内核检查 `sk_receive_queue` 和 `out_of_order_queue` 中当前有多少未被应用取走的数据，从 `sk_rcvbuf` 总量中扣除。
2. **封顶**：即使可用空间很大，`rcv_wnd` 也被 `window_clamp` 封顶。`window_clamp` 通常在连接建立时由 `sk_rcvbuf`、MSS 和窗口缩放因子共同决定。
3. **通告**：每次发送 ACK 时，将当前 `rcv_wnd` 写入 TCP 头部的 16 位 Window 字段（若启用窗口缩放则右移 `rcv_wscale` 位）。

> **注意**：当前文档（`linux-TCB.md`）只列出了字段定义，未展开 `rcv_wnd` 的完整计算流程。以上分析基于字段定义 + 内核 TCP 接收缓冲区管理的通用机制。

### 数据进入预读队列（prequeue）的条件

`prequeue` 是一条**有条件触发的快速路径**。数据不是"自动"进入 prequeue 的——只有当以下条件**同时满足**时才会走这条路径：

```
应用层调用 recv() → 阻塞等待数据
                       ↓
            段到达，序号 == RCV.NXT
                       ↓
            内核检查 ucopy.task != NULL ?
              ↓ YES              ↓ NO
      走 prequeue 直接拷贝    走正常路径
      到用户空间 iov          → sk_receive_queue
```

**必要条件**（从 `ucopy` 结构定义推导）：

| 条件 | 对应字段 | 含义 |
|:---|:---|:---|
| **有进程在等待** | `ucopy.task` | 必须有一个进程已在 recv() 中阻塞（`task != NULL`）——否则没有"用户空间目的地"，无法直接拷贝 |
| **数据是顺序到达的** | — | 段的序号必须等于 `RCV.NXT`（即正好是接收方正期待的下一个字节）。乱序数据只能进 `out_of_order_queue`，不能走 prequeue |
| **用户缓冲区已准备好** | `ucopy.iov` | iovec 数组已填充，内核知道拷贝到用户态的哪个地址 |
| **内存限制未超** | `ucopy.memory` | 每次 prequeue 有内存上限，防止一个 recv 调用吞噬过多内核内存 |

**为什么要搞 prequeue？** 正常的接收路径是：段到达 → 内核将 skb 放入 `sk_receive_queue` → 唤醒阻塞的进程 → 进程从内核态拷贝数据到用户态。这条路径有两次上下文切换（进程阻塞唤醒 + 拷贝时可能缺页）。

prequeue 的优化在于：当进程已经在 recv 中等待时，段到达的软中断处理上下文中直接就能访问用户态地址（`ucopy.iov`），于是可以**在软中断上下文中一次性完成拷贝**，数据直接进入用户缓冲区，完全跳过 `sk_receive_queue` 中转——节省了一次内核队列操作和唤醒开销。这就是"VJ 风格"优化（Van Jacobson 在 BSD 中首创）。

### 乱序队列（out_of_order_queue）的进入条件

你的直觉是对的——并不是所有收到的数据都进乱序队列。数据在放入 `out_of_order_queue` 之前，必须通过 RFC 793 定义的一道"安检门"。

**第一步：序列号可接受性测试（RFC 793, p.25）**

这是 SEGMENT ARRIVES 处理的第一步，也是最核心的过滤条件。接收方判断一个段是否"合法"的四段测试：

| 段长度 | 接收窗口 | 可接受条件 |
|:---|:---|:---|
| SEG.LEN = 0 | RCV.WND = 0 | `SEG.SEQ == RCV.NXT`（只接受刚好落在左沿的纯 ACK） |
| SEG.LEN = 0 | RCV.WND > 0 | `RCV.NXT <= SEG.SEQ < RCV.NXT+RCV.WND` |
| SEG.LEN > 0 | RCV.WND = 0 | **不可接受**（零窗口时拒绝所有数据段） |
| SEG.LEN > 0 | RCV.WND > 0 | `RCV.NXT <= SEG.SEQ < RCV.NXT+RCV.WND` **或** `RCV.NXT <= SEG.SEQ+SEG.LEN-1 < RCV.NXT+RCV.WND` |

第四种情况最复杂：段的起点或终点至少有一个必须落在窗口内。也就是说，即使段的开头已经超出了窗口右沿，只要有一段尾巴落入窗口，仍然是可接受的（开头在窗口外的部分会被截断丢弃）。

**不符合上述任何条件的段 → 直接丢弃并回复一个 ACK（告知正确的 RCV.NXT）。**

**第二步：判断数据路径**

通过序列号检查后，根据 `SEG.SEQ` 与 `RCV.NXT` 的关系分流：

```
SEG.SEQ == RCV.NXT ?
  ├── YES → 顺序段
  │   ├── 有进程在 recv 阻塞? → YES → prequeue 直接拷贝到用户空间
  │   └── 没有进程等待 → sk_receive_queue
  └── NO (SEG.SEQ > RCV.NXT) → 乱序段 → out_of_order_queue
```

**第三步：去重检查**

即使通过了前两步，如果新段的全部字节范围 `[SEG.SEQ, SEG.SEQ+SEG.LEN)` 已经被之前收到的段完全覆盖（已在 `receive_queue` 或 `out_of_order_queue` 中），这个段是重复的。仍然会回复 ACK（可能是 D-SACK），但数据不会进任何队列。

**第四步：缺口填补——从乱序到顺序**

这是理解 `out_of_order_queue` 最关键的一环。当缺失的段最终到达（`SEG.SEQ == RCV.NXT`）：

1. 该段直接放入 `sk_receive_queue`（或走 prequeue），推进 `RCV.NXT`
2. 内核扫描 `out_of_order_queue`，找出所有与新的 `RCV.NXT` **连续**的段
3. 将这些连续段**批量转移**到 `sk_receive_queue`，继续推进 `RCV.NXT`
4. 重复直到遇到下一个缺口

这意味着乱序队列中的数据并非"等着被重传"——它们**已经在了**，只是前面的缺口让 `RCV.NXT` 无法推进。一旦缺口补上，所有后续的连续数据瞬间变为可用。

**完整流程总结**：

```
段到达
  │
  ├─[1] 序列号可接受性测试 (RCV.NXT ≤ SEG.SEQ < RCV.NXT+RCV.WND?)
  │      └── 不通过 → 丢弃，回复 ACK
  │
  ├─[2] RST 检查 → 重置连接
  ├─[3] 安全/优先级检查
  ├─[4] SYN 检查 → 已建连接收到 SYN 则 RST
  ├─[5] ACK 处理 → 推进 SND.UNA，释放发送缓冲区
  ├─[6] URG 处理 → 标记紧急数据
  │
  ├─[7] 处理段文本:
  │      ├── SEG.SEQ == RCV.NXT → 顺序: receive_queue / prequeue
  │      └── SEG.SEQ > RCV.NXT  → 乱序: out_of_order_queue
  │
  └─[8] FIN 检查 → 推进状态机
```

**一个具体的例子**：

假设 `RCV.NXT=1000`, `RCV.WND=5000`，接收方已收到段 `[2000, 3000)` 和 `[3500, 4000)`（在 out_of_order_queue 中）。现在段 `[1000, 2000)` 到达：

1. 通过序列号检查（SEG.SEQ=1000 == RCV.NXT） ✓
2. 是顺序段 → 放入 receive_queue
3. `RCV.NXT` 推进到 2000
4. 扫描 out_of_order_queue → 发现 `[2000, 3000)` 连续 → 转移 → `RCV.NXT` 推进到 3000
5. 发现 `[3000, 3500)` 缺失 → 缺口 → 停止
6. `RCV.NXT` 最终停在 3000，应用可以立即读取 `[1000, 3000)`

### 从现有文档看 TCP 拥塞控制

> **重要声明**：当前文档库中不包含完整的拥塞控制算法规范。RFC 793（1981）早于拥塞控制的发明（Jacobson 1988）。`linux-TCB.md` 列出了拥塞控制相关的**字段名和状态名**，但没有展开算法流程。以下分析完全基于现有文档片段可验证的内容。

---

**现有文档中出现的拥塞控制字段**（`linux-TCB.md` L92-L117）：

| 字段 | 推测含义（从字段名推断） | 文档是否有详细解释 |
|:---|:---|:---|
| `snd_cwnd` | 发送拥塞窗口——发送方自己维护的"不超网络承载能力"上限 | 仅字段名，无算法 |
| `snd_ssthresh` | Slow Start Threshold——慢启动阈值，cwnd 增长的转折点 | 仅字段名，无算法 |
| `packets_out` | 飞行中的数据包数（已发但未确认） | 仅字段名 |
| `retrans_out` | 正在重传的数据包数 | 仅字段名 |
| `lost_out` | 判断为丢失的数据包数 | 仅字段名 |
| `sacked_out` | 被 SACK 选择性确认的数据包数 | 仅字段名 |

**拥塞控制状态**（`ca_state`，Linux `tcp_sock`）：

| 状态 | 文档中出现 | 可确认的含义 |
|:---|:---|:---|
| **Open** | `linux-TCB.md:92` | 正常状态，未检测到丢包，按当前算法增长 cwnd |
| **Disorder** | `linux-TCB.md:92` | 检测到轻微的乱序（收到 dupACK 但不够 3 个），发送方开始警觉但尚未行动 |
| **CWR** | `linux-TCB.md:92` | Congestion Window Reduced——收到 ECN 拥塞通知后正在缩减窗口 |
| **Recovery** | `linux-TCB.md:92` | 快速恢复——收到 3 个 dupACK，已重传丢失段，正在从丢包中恢复 |
| **Loss** | `linux-TCB.md:92` | RTO 超时——严重拥塞，窗口将重置 |

**现有文档能确认的关键事实：**

1. **"两窗取小"** — RFC 793 定义了 rwnd（接收方通告窗口），加上后来引入的 cwnd（拥塞窗口）。实际可发送字节数 = `min(cwnd, rwnd)`。这个机制在 linux-TCB.md 的字段定义中得到了印证（`snd_cwnd` 和 `snd_wnd` 是两个独立字段）。

2. **RTO 超时重置 cwnd** — 知识库中提到"RTO 后拥塞窗口重置为 1，进入慢启动"。这是拥塞控制最极端的手段：当超时发生时，发送方假定网络发生了严重拥塞，毁灭性地把 cwnd 打回 1。

3. **SACK 保留拥塞控制** — RFC 2018 明确写道"fast recovery per ACK max 1 segment (Reno style) or 2 (slow start)"。SACK 不改变 cwnd 的增长规则，只改变"重传哪些段"的决策。

4. **dupACK 阈值可调整** — RFC 2883 (D-SACK) 提到可以用 D-SACK 信息"调整 dupACK 阈值"或"撤销 cwnd 缩减"，暗示了 Fast Retransmit 的标准触发条件是收到 3 个 dupACK。

5. **可插拔拥塞算法** — `linux-TCB.md` 显示 Linux 2.6.12 已支持三种拥塞控制算法：Vegas（基于 RTT 变化的预测）、Westwood（基于带宽估计）、BIC（Binary Increase Congestion，Linux 默认算法）。每种在主 TCB 中有独立的结构体。

6. **ECN 支持** — `linux-TCB.md` 包含 `ecn_flags` 字段，说明 Linux TCB 支持显式拥塞通知（RFC 3168）——路由器可以在不丢包的情况下标记拥塞。

---

**文档缺口**：以下内容是 TCP 拥塞控制的核心，但当前文档库中没有覆盖。如果需要精确理解，建议补充 RFC 5681（定义慢启动/拥塞避免/快速重传/快速恢复）：

- 慢启动（Slow Start）的具体算法（cwnd 指数增长直到达到 ssthresh）
- 拥塞避免（Congestion Avoidance / AIMD）的具体公式
- 快速重传（Fast Retransmit）的精确触发条件和行为
- 快速恢复（Fast Recovery）中 cwnd 和 ssthresh 的具体更新规则
- RTO 退避（exponential backoff）的详细算法

## 常见误解快查（可选）

- ❌ "SACK 和普通 ACK 是两套独立的确认机制" → ✅ SACK 是累积 ACK 的**补充**，不改变累积 ACK 字段的含义和权威性。累积 ACK 永远是"正式确认"，SACK 只是"额外信息"。
- ❌ "D-SACK 是 SACK 的升级替代" → ✅ D-SACK 是 SACK 的**扩展**，不是替代。它只是规范了"当收到重复段时 SACK 第一个块应该怎么填"，核心机制不变。
- ❌ "SACK 块越多越好" → ✅ TCP 选项区总共才 40 字节，SACK 块越多留给其他选项的空间越少。实际使用中经常只有 3-4 个块的空间（与 Timestamp 等其他选项共享）。
