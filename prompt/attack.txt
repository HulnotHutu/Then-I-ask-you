# “头脑风暴" 模式：
- 你现在要说服我TCP就是需要进行三次握手，必须遵守 SYN（seq，ack=0）、SYN+ACK（seq，ack=seq+1）、ACK（seq=ack+1，ack=seq+1），我会不断发出质疑，你都必须说服我，并且给我理由。
```
1. 我对你的”服务端看到一个新的 SYN，会认为客户端又发起了一次新连接（因为它不知道这是过期的请求）。于是服务端回复 SYN+ACK(seq=z, ack=x+1)，并分配资源（缓冲区、控制块）等待客户端的 ACK。“的说法进行反驳，服务端其实有条件进行拒绝，首先它packet可以携带源 IP 和 目标 IP 、源 port  和 目标 port，针对重发的 packet 同一个客户端同一个进程发送的 源 IP 和源 port 肯定是一样的，然后是发到服务器端的进程，那么目标 ip 和 目标 port 也是一样的，那么我可以对相同的 packet 进行拒绝，那么就是只有一个连接，服务端也不会因为重发的客户端的 SYN 去发送 SYN+ ACK，所以你的说法不对。
2.我对你的 ”服务端看到一个新的 SYN，会认为客户端又发起了一次新连接（因为它不知道这是过期的请求）。于是服务端回复 SYN+ACK(seq=z, ack=x+1)，并分配资源（缓冲区、控制块）等待客户端的 ACK。“的说法进行反驳，客户端不会分配接收的缓冲区，那么服务器在发送是数据时，也是有 seq 的，那么即使不是 SYN 数据包，那么客户端看到服务器的 seq 之后，那么也能分配资源用来接收 服务器发送的数据，
3. 我对你的”服务端却以为连接已建立，开始向客户端发送数据。这些数据会被客户端丢弃（因为没有对应的接收窗口），服务端永远收不到确认，最终超时重传，浪费带宽和 CPU。“的说法发出反驳，确实服务器发送数据会因为没有正确建立连接而丢弃，但是即使收到一个数据包，那么也能按照我第2个反驳的猜想，让客户端接收到数据。
4. 你说的可靠传输到底怎样才是可靠？
```
> 这个需要确认一个论点是正确的，而你对这个存在独特的见解|
